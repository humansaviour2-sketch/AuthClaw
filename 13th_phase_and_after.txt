Implementation Plan - Phase 13: Enterprise Readiness & E2E Verification
This revised plan addresses the transition from Phase 12 validation to Phase 13 integration testing and hardening. It provides a detailed gap analysis of the current test suite, lists unfinished original tasks, introduces new core deliverables (Audit Inspector, Chat Session Memory), and designs the upcoming enterprise RAG architecture and cloud integration roadmap.

User Review Required
IMPORTANT

Chat Memory State Schema: We are introducing database tables chat_sessions and chat_messages in PostgreSQL with row-level security (RLS) to support session history and tenant-scoped chat memory.
RAG Database Addition: The design proposes Qdrant as our enterprise vector database, operating in a multi-tenant payload-filtered model to ensure absolute resource isolation.
Playwright Suite: Playwright end-to-end browser tests will be introduced in the console directory to validate frontend dashboard, audit explorer, and workflow HITL screens.

POST /v1/workflows/chat/sessions/{id}/message — Post a user message, process it with LangGraph memory state, save the agent's reply, and return the response.

3. Post-Phase-13 Roadmap: Real Cloud Connectors
To transition the orchestrator's worker from mock connectors to production-ready integrations, the following roadmap is proposed:

Mermaid diagram
AWS Integration
Credentials: Implement AWS STS AssumeRole in AWSConnector using tenant AWS account ARNs.
Scan: Call boto3 APIs (e.g. s3_client.get_public_access_block(), iam_client.generate_credential_report()).
Remediation: Call s3_client.put_public_access_block() or generate a Terraform plan showing bucket lock configurations.
Azure Integration
Credentials: Use Azure Active Directory Service Principal or Managed Identities.
Scan: Query SQL databases and Network Security Groups via azure-mgmt-sql and azure-mgmt-network.
Remediation: Reconfigure firewall configurations using the management library.
GCP Integration
Credentials: GCP Workload Identity Federation or Service Account JSON keys stored in HashiCorp Vault.
Scan: Check IAM policy bindings via google-cloud-iam or bucket access control via google-cloud-storage.
Remediation: Call IAM set_iam_policy() API to assign roles/viewer in place of roles/owner.
4. RAG Architecture Design (Design Only)
We propose the following enterprise RAG architecture to support regulatory search and evidence indexing:

Mermaid diagram
Key Components
Vector Database (Qdrant):
Compliance documents (GDPR articles, HIPAA safeguards) are parsed and chunked.
Text chunks are converted to vector embeddings using Gemini's text-embedding-004 API.
Vectors are stored in Qdrant.
Multi-Tenant Isolation (Logical Namespacing):
To prevent tenant data cross-contamination without spawning hundreds of vector collections, a metadata payload filter model is enforced.
Every vector point contains a payload property "tenant_id": "<tenant-uuid>".
When searching, the application forces a filter:
json

{
  "filter": {
    "must": [
      { "key": "tenant_id", "match": { "value": "a0eebc99-0000-0000-0000-bb6d6bb9bd11" } }
    ]
  }
}
Audit Log Hook:
Every document ingestion, vector upload, or semantic search query generates an audit event published to Kafka.
The event contains the hash of the document/query and is immutably stored in ClickHouse.
RAG Introduction Point
Where: Introduce RAG inside the LangGraph orchestrator node ANALYZE_COMPLIANCE (currently mock-analyzed).
Execution Flow:
The scanning agent detects security findings from cloud metadata.
The agent queries Qdrant (filtered by the user's tenant_id) to find the exact compliance articles (e.g. GDPR Art.25 for AWS S3) matching the findings.
Relevant regulatory chunks are appended to the context window of Gemini to generate exact, compliant remediation advice and link logs to regulatory controls.
